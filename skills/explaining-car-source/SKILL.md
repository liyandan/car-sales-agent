# 车源档案的基础信息及车型选装、配置信息讲解

## 使用场景

本 Skill 用于为用户讲解二手车基础配置及亮点信息。

---

## 工具脚本概览

所有脚本位于 `scripts/tools/` 目录下，通过 `python scripts/tools/<script>.py` 调用，返回 JSON 格式结果。

## 车辆档案讲解流程

核心特征：结论先行、一句到位、用数据而不用术语、把参数翻译成用车场景。

---

### Step 1 — 理解意图

快速判断用户问的是什么。问题意图明确时直接进入 Step 2；问题模糊时用一句话确认，例如"您是想了解这车的电池状况对吧？"，确认后再继续。不做多余寒暄。

---

### Step 2 — 查询数据

根据用户问题类型，从上方脚本索引表中选择对应脚本，**加载其 reference 文件**确认入参与返回字段，然后执行脚本获取数据。

**执行要点：**

1. 从上下文中提取 `clue_id`（必填）和 `env`（必填，不可缺省）。
2. 查询车源所在城市时，额外提取 `plate_city_id`、`selected_city_id`、`location_city_id`。
3. 无车源的通用配置查询使用 `--question` 参数代替 `--clue_id`。
4. 从返回的 JSON 中锁定与用户问题直接相关的字段，忽略无关字段。

**数据状态判断：**

- 查询到有效数据 → 进入 Step 3。
- 数据缺失或无法回答 → 进入 Step 4。
- 查到「车辆在售状态」为 `已售出` 或 `已下架` → 直接告知用户车已售出（如"不巧，这车刚被订走了"），不再继续讲解。

---

### Step 3 — 组织回答

按「结论 → 关键数据 → 场景价值」的节奏输出回复，三者可按需取舍，但结论永远在最前面。

**3.1 结论先行**

用一句话给出核心判断，抓住注意力。

> "电池状态很健康" / "配置非常丰富"

**3.2 数据佐证**

用具体数字支撑结论，将专业术语翻译为用户能理解的表达（如 SOH → 电池健康度）。

> "电池健康度97%，参考续航543公里，还有8年/16万公里三电质保"

对于配置类问题，按模块分类（安全 / 舒适 / 智能 / 动力续航 / 辅助驾驶）简要列出核心卖点，避免一次性罗列全部参数。

**3.3 场景化收尾（可选）**

当某项配置或参数对用户有明显价值时，用一句日常场景帮用户感知：

> "高速上开自适应巡航加车道保持，长途很省心"
> "这种小车就是城市代步，实用为主"

**对于「异常点」的处理：** 将高里程、多过户等潜在顾虑与车辆定位结合进行合理化解释。例如商用车高里程属于正常使用场景；过户次数可结合车主换车、资产调配等常见情况说明。

---

### Step 4 — 透明引导

当查询数据缺失或不足以回答时，坦诚说明现状并给出可操作的验证路径，不编造信息。

**节奏：说明现状 → 给出方案**

> "这项信息暂时查不到，您可以到店看实车确认"
> "您可以在详情页查看完整参数"

可引导的验证路径包括：查看详情页、到店实车确认、平台专业检测等但不要引导用户到直播间看车，更不能提及与时间相关的引导语。

---

### Step 5 — 输出与卡片提取

完成回答内容后，按以下规则生成最终输出：

**回复文本要求：**

- 控制在 50 字以内（需提供链接的情况除外）。
- 不包含 emoji 符号，不使用 markdown 格式。
- 删除所有数据来源说明性文字（如"根据查询结果"、"从系统返回来看"、"根据检测报告"等），直接给出答案。
- 已提取为卡片的链接须从正文中移除。
- 参考对话历史、用户偏好等上下文信息，确保回答与用户已知信息衔接一致。
- 当工具返回空值、异常信息或数据缺乏参考价值时的处理原则请查阅 [references/error_processing_rule.md](references/error_processing_rule.md)
- 当有卡片信息提取时，不进行所提取卡片以外的信息引导，尤其不进行直播间看车引导

**卡片提取：**

扫描回复正文及工具调用结果，若涉及车辆图片查看链接（整车/外观/内饰/机舱工况）且已经获取到有效链接，需要提取为 `mini_program`（小程序）消息卡片并写入 `message_card`。无相关链接时 `message_card` 为空列表。
# 消息卡片提取与转换
## 触发条件

当满足以下**任一**条件时，必须执行卡片提取流程：

1. 用户明确要求查看：整车图片、全车图片、外观图片、内饰图片、机舱工况图片，或上述内容的链接。
2. 回复正文（`text` 字段）或工具调用结果中包含了上述类型的图片/链接 URL。

---

## 提取流程

1. **识别链接**：扫描用户问题、回复正文及工具调用返回结果，找到与图片查看相关的唯一最匹配 URL。
2. **删除正文中的链接**：将 `text` 字段中对应的原始 URL 移除，不在回复正文中展示裸链接。
3. **构建卡片对象**：将提取到的链接按下方格式转换为结构化卡片数据。
4. **写入 `message_card`**：将卡片对象放入列表，作为最终输出 JSON 的 `message_card` 字段。

---

## 卡片格式

卡片类型固定为 `mini_program`（小程序卡片），列表中**仅包含一个**卡片对象。

```json
{
  "type": "mini_program",
  "clue_id": <int>,
  "card_type": <int>,
  "title": "<string>",
  "url": "<string>"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `type` | string | 是 | 固定值 `"mini_program"` |
| `clue_id` | int | 是 | 当前会话/车源对应的实际 clue_id，禁止使用占位或示例数字 |
| `card_type` | int | 是 | 固定值 `1001`（适用于外观、内饰、机舱工况、整车、全车图片） |
| `title` | string | 是 | 根据用户实际提问内容确定，取值规则见下方 |
| `url` | string | 是 | 从回复正文或工具调用结果中提取的、与 title 对应的唯一链接 URL |

---

## `title` 取值规则

| 用户提及内容 | `title` 取值 |
|-------------|-------------|
| 外观（图片） | `"外观图片"` |
| 内饰（图片） | `"内饰图片"` |
| 机舱工况（图片） | `"机舱工况图片"` |
| 全车（图片） | `"全车图片"` |
| 整车（图片） | `"整车图片"` |
| 外观 + 内饰（两项组合） | `"外观/内饰"` |
| 外观 + 内饰 + 机舱工况（三项全部） | `"内饰/外观/机舱工况"` |
| 其他两两组合 | 按用户提及的类型以 `/` 拼接 |

> **注意**：「外观图片」与「整车图片」是不同概念。用户提问外观图片时，`title` 必须使用 `"外观图片"`，不得替换为 `"整车图片"`。

---

## 排序与禁止事项

- 多个候选链接时，以链接在「用户问题 → 回复正文」中**首次出现的顺序**排列，最终仅保留最相关的一个卡片对象。
- 禁止为未在正文或工具调用结果中实际出现的链接编造卡片。
- 禁止将非链接内容误识别为需要转换的条目。
- 无任何需要转换的链接时，`message_card` 输出为空列表 `[]`。

**最终输出格式：**

```json
{
  "data": "回答依赖的核心数据摘要（工具返回超200字符时进行精简）",
  "text": "最终回复内容",
  "message_card": []
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `data` | string | 是 | 最终回答所依赖的核心关键数据。若工具返回数据超过 200 字符，应进行摘要处理后填入 |
| `text` | string | 是 | 最终回复内容。已提取为卡片的链接须从正文中移除 |
| `message_card` | array | 是 | 提取到的卡片对象列表。无卡片时为空列表 `[]` |
